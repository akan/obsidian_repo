过去几年，法律科技公司最值钱的东西不是模型，也不是产品体验，而是“律师不会写代码、程序员不懂法”这条信息差，这导致一些法律科技公司在早期有勇气卖出高溢价产品，尽管这些产品并不算趁手。

  

但 Vibe Coding 出现后，这条鸿沟正在被快速填平。去年底，我用Anti Gravity ，花了半天迭代，做出了一个能在 ".docx" 里自动批注和修订合同的工具，这意味着法律工具正在从“购买软件”变成“按需构建”，下文我会结合一次亲手实践，解释这条护城河为什么正在被填平。

  

> 如果说LLM（大语言模型）为普通人打破了知识的瓶颈，那么Vibe Coding则是进一步突破了执行的壁垒。

  

  

## 什么是Vibe Coding

所谓 **Vibe Coding** ，直译过来或许可以叫“氛围编程”。这个词最早由特斯拉前 AI 总监 Andrej Karpathy 带火 。它的核心理念非常简单： **把“写代码”变成“描述目标 + 迭代验收”** 。

过去 AI 也会写代码，但它更多停留在聊天框里；运行、测试、迭代仍然麻烦，Vibe Coding 则把这些环节整合成了一个连续工作流。

  

## 律师的Vibe Coding

在两年前我非常希望有内地厂商去复刻Spellbook这样的合同审查工具，它直接工作在Word侧栏，会把审查意见嵌入".docx"文档作为批注和修订显示，这让AI交付给律师的成果更接近律师给客户交付的形态，换句话说， **离结果更近** 。

  

今天我们来看当下的内地AI合同审查工具，大概率会发现如下问题：

1. 1\. 为满足合规要求，往往 无法提供最好的模型
2. 2\. 审查 流程难以自定义 ，且对用户而言不透明
3. 3\. 交付结果一言难尽：比如为了多平台兼容性，有的产品选择了Web应用，但审查结果 没法导出或者格式存在缺陷 ，有的产品为了适配模型的上下文，强行将文档拆分后进行审查，导致 合同出现一致性问题 ，有的产品选择了参考合同模板进行审查，却 不能做到合同的准确匹配，并且参考合同本身的时效性、可靠性也不能保证......
4.

就我个人的经验而言，AI合同审查，大部分情况下 **由AI自身的能力决定成果上 限** ，在过去，选一个顶级模型，让其输出风险和修改意见的表格，然后边核查内容边修订".docx"文档，整个过程已经相对高效，但在量的堆积下，修订和批注的工作仍然会让人感到疲惫。于是我尝试更进一步，在AI审查的基础上进一步实现批注和修订的自动化。

### 这里介绍实现过程的一些关键点

2025 年上半年，Claude Code 让 AI 住进了电脑，当时我也曾尝试解决这一问题，虽然看起来简单，但做起来却很复杂，AI推荐的方案是 **直接生成一个修订后的版本，然后做对照来实现修订红线** ，但测试后发现，想要稳定生成“修改”后的全文并不容易，再加上我对命令行的使用并不熟练，因此未能跑通。

2025 年底，Google 的 Anti Gravity 进一步降低了交互门槛，让更多非程序员能把“想法”打造为“工具”，这一次我也转变了思路， **尝试借鉴SpellBook的经验** ，于是我从网上找来了它的部分代码，让AI分析出它是如何实现批注与修订的，我几乎看不懂这些代码的内容，只是把这些代码都丢给Anti Gravity去理解和分析，但AI确实告诉我SpellBook用了一个聪明的办法。

接下来，我在同一个对话框里让Anti Gravity按照它的经验，构建一个完整的合同审查应用。

AI生成第一个版本的时间很快， **只用了5分钟不到** ，随后我用了大约一下午的时间来反馈问题和改进，最终就得到了这样一个工具。

只要在设置中配置好API，选择文档后，它会自动检测合同方，然后在侧栏生成审查结果的预览。

完成后点击应用到文档，就会生成一个reviewed版本的".docx"文档。

更重要的是，这套工具可以 **完全按团队和客户的工作方式来定制** ：

比如重点盯某类特定风险、固定表达方式、套用审查模板、或者在修订稿之外额外输出一份审查意见。

这些能力是 **我们自己用代码就能快速实现** 的。

市面上的通用产品很难真正适配这种高度个性化的需求，但自己写的工具，只需要几轮对话就能迭代出你需要的版本。

**这种“细到每一步流程”的灵活性，是通用软件永远无法企及的。**

虽然我不认为程序员做的工具会立刻替代法律人的专业判断，但它确实能向企业交付成果，而这是才是核心。

对企业，特别是初创企业来说，如果他们需要的是 **降低成本 + 控制大方向风险** ，“80分可用的自动审查”或许就足够了，而AI在满足这一需求上有无可撼动的成本优势， 况且头部AI也不只能做到80分而已。

  

## 从购买服务到构建能力

> 律师和程序员都在试图把对方“封装”起来，而法律科技公司的处境就显得微妙了。

### 对律师而言

Vibe Coding 的出现，不仅填平了法律科技公司浅层的护城河，更重要的是，它 **归还了法律人对工具的定义权** 。未来的律师，可能不仅仅需Know-how，而更需要懂得如何将自己的Know-how封装进工具，并让其规模化运行。

### 对企业而言

LLM+Vibe Coding，让法律服务变得更加触手可得，他们可直接 **面向需求制造工具** ，最大化发挥AI带来的红利，这对于提供法律服务的群体来说恐怕不是个好消息。

### 对法律科技公司而言

这是一种双向的侵蚀。一边是律师在丢弃他们的工具，另一边是企业在丢弃他们的服务，如果他们的产品依然只是简单的套壳，或者是隔靴搔痒的通用SaaS，那确实该慌了。

**现在懂痛点的人和真需求的人都已经拿起了锤子，当剑士们不依赖铁匠就能自己造出趁手的兵器时，谁还会去买那些不合手的标准化工具呢？**

当然，我并不是说所有 LegalTech 都没有价值。真正有壁垒的公司，仍然能依赖行业 Know-how、深度交付能力，获取忠实的用户。但如果产品的核心能力只是套壳，那确实会被 Vibe Coding 迅速抹平护城河。
