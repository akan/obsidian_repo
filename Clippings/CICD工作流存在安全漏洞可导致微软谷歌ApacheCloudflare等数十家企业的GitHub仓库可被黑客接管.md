重点摘要

本次 CI/CD 安全风险的根源，不在于工具存在底层漏洞，而是长期存在的安全认知偏差。GitHub Actions 的 YAML 流水线并非简单静态配置，它支持执行系统命令、托管核心密钥、驱动线上版本发布，拥有极高操作权限，但绝大多数开发人员未将其视作核心业务代码管理，编写、修改完全随手操作，缺少评审与安全校验。 当外部陌生人仅通过提交 PR、留言评论就能无门槛触发整套高权限流水线时，企业代码仓库的安全边界已经彻底失守，相当于对外敞开了入侵通道。

## 01 Cordyceps 是什么？

它不是某一款工具独有的程序漏洞，而是 **CI/CD 工作流搭配不当形成的完整攻击链路** 。单独看每一段配置、每一行代码都完全符合官方规范，不存在语法报错，但组合在一起就打通了一条入侵通道：外部匿名用户输入 → 自动执行代码 → 读取核心密钥 → 篡改线上发布包。

漏洞不在某一个工具里，而是出在 **信任边界** 上——不受信任的外部数据，毫无阻拦地进入了高权限自动化流程。所有组件本身没问题，错的是开发者把它们拼凑在一起的方式。

## 02 危害极大，攻击门槛极低

**攻击者零门槛操作，** 只要注册一个公开免费 GitHub 账号，不需要加入企业组织、不需要仓库读写权限，就能发起攻击，伪造审批流程、窃取永久密钥、推送篡改代码。

根源在于绝大多数开发者存在认知误区：大家把 GitHub Actions 的 YAML 工作流文件当成简单配置，随便编写、很少审核。但实际上这类文件可以执行系统命令、读取加密签名密钥、登录云服务器、对外发布线上版本， **本质是具备高权限的业务代码** ，却长期没有纳入安全管控，这就给攻击者留下了巨大突破口。

Novee 团队批量扫描了 npm、PyPI、Go 等主流生态三万余个高影响力仓库，先用自动化工具筛出可疑配置，再通过 **AI 智能体模拟攻击验证** ，最终确认 300 多个仓库能被完全接管。因为这是一类通用配置漏洞，不是单一软件漏洞，全球潜在受影响仓库数量 **可达数百万** 。

## 03 传统安全扫描工具为什么完全查不出来？

市面上常规代码扫描工具，只会逐文件检索已知漏洞特征，判断单文件是否存在风险。但 Cordyceps 的风险藏在 **多流程联动逻辑** 里，单份 YAML 文件语法完全合规，工具无法识别"外部 PR 可以触发高权限流程"这条隐藏攻击链。

ATTACK CHAIN · 完整攻击链路

① 陌生外部用户提交 PR / 留下评论

② 平台自动触发仓库自带高权限 CI 流水线

③ 攻击者恶意代码在官方构建服务器运行

④ 读取流水线内置密钥、云账号令牌

⑤ 窃取凭据篡改代码、发布恶意版本 → 仓库彻底失守

整条链路每一步都符合平台配置规则，唯独所有配置流程组合到一起没人做安全审查，管控"外部不可信数据是否能进入内部权限流程"，这也是传统安全扫描工具集体"失明"的核心原因。

## 05 漏洞底层原理来自于老风险的规模化爆发

安全圈内很早就熟知 **"pwn request"** 这类 CI/CD 风险，核心诱因是 `pull_request_target` 触发器。该触发器会携带主仓库完整密钥、访问令牌运行流水线；如果工作流中自动拉取并执行外部分支代码，外部攻击者就能直接拿到全套高权限。

同类安全事件早已多次爆发：

· **tj-actions** 漏洞波及 2.2 万个上下游仓库

· **Nx** 构建系统恶意投毒事件，连累 PostHog、TanStack 等知名项目

· **Ultralytics、Trivy** 等开源项目曾被植入挖矿程序

行业调研显示， **38% 的企业** CI/CD 配置存在命令注入、危险触发器问题。

而 Cordyceps 事件的关键区别在于：安全团队把零散个案 **做成了规模化检测手段** ，依靠自动化 + AI 批量扫描验证，证明这是全行业普遍存在的共性问题。

## 06 AI 辅助开发（Vibe Coding），正在放大这个隐患

如今绝大多数开发人员会借助 AI 工具快速生成 CI/CD 配置文件，但 AI 的训练素材来源于网络上大量不安全的旧模板，会 **持续复制同款高危写法** 。

安全工程师的总结

依靠 AI 智能体写代码的模式，会让这类 CI/CD 漏洞 **指数级扩散** ，大量仓库重复踩同一个坑，风险持续蔓延。

GitHub 同步推出官方防护

6 月 18 日上线 **actions/checkout v7** 版本，默认拦截最容易被利用的 fork 分支 PR 检出逻辑；7 月 16 日会把防护策略同步到所有主流版本。使用 `@v4` 这类浮动版本标签会自动更新获得防护；如果固定填写具体 commit 哈希值，需要开发人员手动升级。

**⚠️ 注意官方修复存在局限：** 仅阻断"拉取外部 PR 代码合并"这一类场景。通过 `issue` 评论触发流水线、用 `git` 命令手动拉取外部代码、引入第三方仓库等场景，依旧存在风险，不能完全依赖平台兜底。

## 08 企业落地防护方案

**核心准则：** 所有来自外部分叉仓库的 PR，统一判定为不可信输入，严格隔离权限。落地分为 5 条实操规范：

**① 工作流即代码** — 将 CI/CD YAML 等同业务代码管理，纳入代码评审、定期安全审计，禁止随意修改配置。

**② 最小权限** — 流水线、访问令牌仅分配业务必需权限，能只读绝不开放写入。

**③ 来源校验** — 高权限流水线增加仓库来源校验，仅放行本仓库内部人员提交的 PR（见下方代码）。

**④ 两段式工作流** — 外部不可信代码仅在无权限环境运行、产出结果；涉及代码提交、版本发布等高权限操作，单独启用第二条独立流水线。

**⑤ 锁定依赖 + 净化输入** — 所有 Action 依赖锁定完整哈希值、杜绝浮动版本；禁止把 PR 标题、分支名等外部输入直接拼接进 Shell 命令。
