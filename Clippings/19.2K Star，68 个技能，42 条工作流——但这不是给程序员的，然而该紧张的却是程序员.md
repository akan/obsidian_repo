朋友们，我发现并且思考了一件让我有点意外的事。似乎看到了些什么，似乎有看不太清，算是有点点思路吧。

过去三个月，我拆了 ECC（206K Star）、Superpowers（210K Star）、gstack——全是 Agent 技能框架，全是给开发者用的。我在之前的文章里下过一个判断： **Agent 不缺大脑，缺的是操作系统。方法论正在被编译成基础设施。**

我当时觉得这事离普通程序员最近，离其他职业还挺远的。

![Image](https://mmbiz.qpic.cn/sz_mmbiz_png/sc9vl5waw2o2LKVqTibgRSlNg9WEdYr38rXCDpcYiaDR2bjWQgibLDDvAegq2nGnlsPlw0L88j7LutKNzRVlHOC7dG3NzBEuicYJTzlvHWmiaweI/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=0)

然后我看到了 pm-skills。19.2K Star，6 月 5 号刚发 v2.0。它做的事情一句话就能说清楚： **把 Teresa Torres、Marty Cagan、Alberto Savoia 这帮产品大佬的方法论，编码成了 Agent 可执行的技能文件。**

![Image](https://mmbiz.qpic.cn/mmbiz_png/sc9vl5waw2o9D1sSruKTsFTuQCx6JWEU4k7u7qH4Tv43FvEZ1Hug84ZhmhROJeOdn6ia0ZDiaou18pVgYmgWXhFI4iaEbJH216NriaT7S25I2l4/640?wx_fmt=png&from=appmsg&tp=webp&wxfrom=5&wx_lazy=1#imgIndex=1)

不是给程序员的。是给产品经理的。

我花了一个下午把整个仓库翻了一遍。翻完之后，我的感觉不是"PM 也有 Skills 了"，我的感觉说出来你可能并不模式，它是一个更大的信号： **"方法论即基础设施"这件事，正在从程序员圈子溢出到其他领域。**

今天这篇文章，我想聊三件事：pm-skills 到底做了什么，为什么它让我觉得这事不简单，以及这对不在 PM 岗位的你意味着什么，尤其是我生有感触

### pm-skills 到底是个什么东西

先说结构。pm-skills 是一个 Claude Code / Cowork 的插件集合，按 9 个 PM 领域组织：

```
pm-skills/
  pm-product-discovery/     13 个技能，5 个命令（产品发现）
  pm-product-strategy/      12 个技能，5 个命令（产品战略）
  pm-execution/             16 个技能，11 个命令（日常执行）
  pm-market-research/        7 个技能，3 个命令（市场研究）
  pm-data-analytics/         3 个技能，3 个命令（数据分析）
  pm-go-to-market/           6 个技能，3 个命令（上市策略）
  pm-marketing-growth/       5 个技能，2 个命令（增长营销）
  pm-toolkit/                4 个技能，5 个命令（通用工具）
  pm-ai-shipping/            2 个技能，5 个命令（AI 代码交付）
```

68 个技能，42 条链式工作流。作者是 Paweł Huryn，The Product Compass Newsletter 的负责人。

架构和 ECC、Superpowers 一样是三层：

| 层 | 做什么 | pm-skills 的对应 |
| --- | --- | --- |
| Skills | 领域知识 / 分析框架 | 每个 SKILL.md 编码一个 PM 方法论 |
| Commands | 用户触发的链式工作流 | `/discover`  、 `/write-prd` 、 `/strategy` |
| Plugins | 按领域打包的可安装单元 | 9 个 pm-\* 目录 |

看到这张表你可能会说："不就是把 PM 方法论写成 Markdown 吗？" 别急，让我带你看看它到底编码了什么。

### 它编码的不是"怎么写 PRD"，所不同的是，他描述的是"怎么做产品决策"

我就拿三个最有代表性的 skill 说说。

**第一个：Opportunity Solution Tree（产品发现）**

来自 Teresa Torres 的《Continuous Discovery Habits》。你敲 `/discover AI-powered meeting summarizer for remote teams` ，Agent 不是直接帮你写 PRD，他特么的会走一条硬性路径：

```
头脑风暴机会点 → 映射到用户痛点 → 生成解决方案 → 
识别核心假设 → 设计验证实验 → 优先级排序 → 
才进入功能设计
```

每一步都编码了 Torres 的框架逻辑：不要从解决方案出发，要从机会点出发。Agent 会先逼你想清楚"用户到底痛在哪"，然后再往下走。这就是我们常说的，你的用户会在什么情况下使用你这个产品的什么功能，如果你连这个都没想清楚，你产品等于白做了。

**第二个：Pre-mortem（执行阶段）**

来自 Gary Klein 的研究。你敲 `/pre-mortem` ，Agent 会假设"这个项目已经失败了"，然后引导你倒推可能的失败原因，生成一份结构化的风险清单。

这个 skill 厉害的地方不在于它帮你列了什么风险， **在于它强制你做了一件人类天然不愿意做的事：在信心最高的时候想失败。** 行为经济学叫"事前验尸"，pm-skills 把它变成了一个 `/pre-mortem` 命令。

**第三个：/ship-check（AI 代码交付）**

这是我觉得最有意思的一个。它面向的是那些需要为 AI 写的代码负责的 PM 和创始人。你敲 `/ship-check` ，Agent 做两件事：

1. 1\. 生成系统文档（这个系统应该做什么、谁有什么权限、密钥存在哪）
2. 2\. 对比文档和实际代码之间的差距——"文档里说用户数据要加密，代码里加密了吗？"

它解决的是一个非常真实的问题： **AI 写代码很快，但不留下意图记录。** 三个月后没人知道这段代码"应该"做什么。 `/ship-check` 把"意图文档化"变成了一个硬性前置步骤。

是的，你看，这个 skill 即使你不是 PM，也值得跑一下。你让 Agent 帮你写了一堆代码，但你真的知道它写了什么吗？

### 为什么说这是一个信号

好，回到我开头说的那个判断。

过去三个月我拆了十几个 Agent 技能框架，它们全都在做同一件事： **把一个领域里"资深专家脑袋里的方法论"编码成 Agent 可执行的 Markdown 文件。**

> ECC 编码的是软件工程方法论（TDD、Git workflow、安全审查）。Superpowers 编码的是工程纪律（brainstorm → plan → subagent 执行 → code review）。gstack 编码的是创业方法论（Boil the Lake、Search Before Building）。

我想，我的很多朋友都玩过这些仓，安装到过自己的 Claude code 中用过。

那么，现在 pm-skills 编码了产品管理方法论。

你看这四个项目放在一起，一个模式就浮出来了：

| 项目 | Star | 编码的领域 | 方法论来源 |
| --- | --- | --- | --- |
| ECC | 206K | 软件工程 | 社区共建 |
| Superpowers | 210K | 工程纪律 | Jesse Vincent 个人方法论 |
| gstack | — | 创业 | Garry Tan (YC CEO) |
| pm-skills | 19.2K | 产品管理 | Teresa Torres / Marty Cagan / Alberto Savoia |

**我的判断是：当一个领域的核心方法论被编码成 Agent Skills 的时候，这个领域正在经历一次"基础设施化"的拐点。**

软件工程先到了这个拐点——因为程序员天然离代码近，最先意识到"AI 不缺聪明，缺纪律"。现在产品管理也到了。PM 们每天做的事——发现、战略、执行、研究、分析——本质上就是一系列结构化的决策流程。这些流程过去锁在人的脑子里、锁在书里、锁在咨询公司的方法论 PPT 里。现在它们被编译成了 `/discover` 、 `/strategy` 、 `/write-prd` 。

这不是"PM 也用上了 AI"的故事。 **这是"PM 的方法论变成了可执行的代码"的故事。** 这两件事完全不一样。

前者是给 PM 一个聊天框，后者是给 PM 一个操作系统。

### 这件事跟不在 PM 岗位的你有什么关系

你可能会说："我又不是产品经理，这跟我有什么关系？"

三件事。

**第一，你的 PM 同事如果用了这套东西，喂给你的需求文档质量会不一样。**

你想想，一个用 `/discover` 跑过完整发现流程、用 `/pre-mortem` 做过事前验尸、用 `/red-team-prd` 做过红队审查的 PM，写出来的 PRD 和一个拍脑袋写出来的 PRD，差距有多大？对你的开发效率影响有多大？

Skills 框架的溢出效应是： **它会提高整条协作链的输入质量。**

**第二，"方法论编译"这个模式是可迁移的。**

你做的是运维、安全、数据、设计、市场、HR——任何一个需要结构化决策的领域，都可以做同样的事。gstack 的 ETHOS.md 证明了：你不需要写 200 个技能文件，你需要的是 **把你的三条核心原则编码成每次 Agent 调用前都会注入的前置代码** 。

pm-skills 仓库 README 里有一句话说得好：

> "Generic AI gives you text. PM Skills Marketplace gives you structure."

翻译过来： **通用 AI 给你文字，Skills 框架给你结构。** 这个"结构"就是你的方法论、你的纪律、你的判断标准。

**第三，PM Brain 的设计思路值得所有做知识管理的人看看。**

pm-skills 有一个配套项目叫 PM Brain——产品经理的"第二大脑"。它的实现极其简单：纯 Markdown 文件，存在本地。Claude 在回答前先读这些文件，回答后再写入更新。每周五做一次清理。没有向量数据库，没有云端，没有花哨的记忆架构。

```
pm-brain/
  decisions/      # 做过的决策
  research/       # 做过的研究
  metrics/        # 追踪的指标
  meetings/       # 会议纪要
```

你仔细想想，朋友们，这跟我之前做 MemPalace + llm-wiki 的思路高度相似——原文存储、让检索决定什么被唤醒、本地优先。PM Brain 用更朴素的方式验证了同一条路： **你的知识不需要存在 SaaS 里，它可以是你本地的一堆 Markdown 文件，只要你的 Agent 每次都知道去读它。**

### 一个更大的判断

我把 pm-skills 和之前拆过的项目放在一起看，一个更大的图景出来了。

2024 年，Agent 生态在比谁的模型更好、谁的工具更多。2025 年底就开始比谁的 Harness 设计更好。2026 年，比的维度又多了一个： **谁的方法论编码得更好。**

而且这个"编码"正在跨域扩散。先是程序员（ECC、Superpowers），然后是创业者（gstack），然后是产品经理（pm-skills）。下一个会是谁？设计师？运维？法务？我不知道。但我知道方向是对的。

**因为方法论是人类在每个领域里积累的最值钱的东西。当它从"锁在人脑里"变成"编译成可执行代码"，整个领域的效率上限就被重新定义了。**

这不是 AI 替代人。这是人的智慧被编译了。
