## 一、先纠正一个最常见的误解

很多人把 skill 理解成"多写几个 Markdown 提示词"，以为就是给 AI 写一段更详细的说明。

**Anthropic 在文章里反复强调：这是一个误解。**

在 Claude Code 里，skill 是一个 **目录** ，不是一个文件。这个目录里除了 `SKILL.md` ，还可以放：

- **scripts/** —— 可复用脚本和辅助函数
- **references/** —— API 文档、数据参考、代码片段
- **assets/** —— 模板、配置文件
- **examples/** —— 参考样例和示例数据
- **hooks/** —— 运行时动态钩子（比如记录使用日志）

模型不是一次性吞下所有内容，而是 **根据任务需要逐步探索这些文件** 。这意味着 skill 的设计思路不是"把所有信息塞进一大段提示词"，而是"把真正重要的内容拆到合适的文件中，让 agent 在需要时再去读"。

这其实已经不是传统意义上的 **prompt engineering** ，而更接近：

> **context engineering + capability packaging**

Prompt 只是入口，文件系统、脚本和数据结构才是真正拉开效果差距的部分。

## 二、Anthropic 内部总结的 9 类 Skills

Anthropic 的团队把内部所有 skills 做了分类，发现它们可以归纳为 **9 个类别** 。这个分类非常有启发，因为它告诉我们：skill 不只适用于"生成代码"，而是覆盖了 **软件交付的完整链路** 。

### 第 1 类：库与 API 参考（Library & API Reference）

这类 skill 用来告诉 agent： **某个库、CLI 或 SDK 应该怎么正确使用** 。

尤其适合内部工具、私有封装，或者默认文档不完整的系统。它的价值在于减少"看起来会写、实际上容易踩坑"的错误。

**典型例子：**

- `billing-lib` —— 内部计费库的边缘情况、常见陷阱
- `internal-platform-cli` —— 内部 CLI 的每个子命令及使用场景示例
- `sandbox-proxy` —— 开发环境的出口网关配置：哪些主机可达、怎么调试 "connection refused"、怎么添加白名单

这类 skill 通常包含一个 `references/` 文件夹，里面有代码片段和 **gotchas** （最容易踩坑的点）。

### 第 2 类：产品验证（Product Verification）——质量影响最大的一类

这类 skill 描述 **如何测试和验证代码是否真的工作了** 。

原文非常强调这一类 skill，给出了一个 **反直觉的结论** ：

> **"验证 skills 对 Claude 输出质量的可衡量影响最大。值得让一个工程师花整整一周时间把验证 skills 做到极致。"**

这类 skill 通常配合 **Playwright** 、 **tmux** 等外部工具使用，通过自动化测试来验证功能。

**典型例子：**

- `checkout-verifier` —— 用 Stripe 测试卡驱动结账 UI，验证订单状态是否正确落地
- `tmux-cli-driver` —— 针对需要 TTY 的交互式 CLI 测试

**关键技巧：**

- 让 Claude **录屏** 它的测试过程，这样你能看到它到底测试了什么
- 每一步都用 **程序化断言** 验证状态，而不是"看起来对了"
- 在 skill 里放多种测试脚本，覆盖不同场景

### 第 3 类：数据获取与分析（Data Fetching & Analysis）

这类 skill 连接到你的 **数据和监控体系** ，让 agent 能快速定位问题、做分析。

**典型例子：**

- `funnel-query` —— "注册 → 激活 → 付费"需要 join 哪些事件表，以及哪个表里有 canonical user\_id
- `cohort-compare` —— 对比两群用户的留存或转化，标记统计显著差异，链接到 segment 定义
- `grafana` —— datasource UID、集群名、问题到 dashboard 的查找表
- `datadog` —— 字段参考（@request\_id vs trace\_id）、服务列表、指标前缀约定

这类 skill 的精髓是： **把数据查询的"隐知识"变成显式文档** 。比如 "这个请求 ID 在 Datadog 里叫 @request\_id，但在日志系统里叫 trace\_id"——这种知识以前只在老工程师的脑子里，现在写进 skill，AI 也能用。

### 第 4 类：业务流程与团队自动化（Business Process & Team Automation）

这类 skill 把 **重复性工作流变成一条命令** 。

它们通常指令比较简单，但可能依赖其他 skills 或 MCPs。一个关键设计是： **把之前的执行结果保存在日志文件里** ，帮助模型保持一致性并反思之前的执行。

**典型例子：**

- `standup-post` —— 聚合工单系统、GitHub 活动和 Slack 消息 → 格式化的站会更新，只显示变化（delta-only）
- `create-<ticket-system>-ticket` —— 强制 schema（有效的枚举值、必填字段）+ 创建后的工作流（通知审核人、链接到 Slack）
- `weekly-recap` —— 合并的 PR + 关闭的工单 + 部署 → 格式化的周报

这类 skill 不一定最复杂，但往往最容易在团队里 **快速产生效率收益** 。

### 第 5 类：代码脚手架与模板（Code Scaffolding & Templates）

这类 skill 生成 **有强约束的代码骨架** 。重点不是"会不会生成"，而是"生成后符不符合团队规范"。

**典型例子：**

- `new-<framework>-workflow` —— 用你们的注解规范搭建新的 service/workflow/handler
- `new-migration` —— 迁移文件模板 + 常见陷阱
- `create-app` —— 新的内部应用，预配置好认证、日志和部署

这类 skill 特别有用的地方是：你的团队有一些 **默认约定** （比如所有新服务必须用统一的日志格式、认证中间件、监控埋点），但新人不知道。skill 可以把这些约定编码进去，让 AI 生成代码时自动遵守。

### 第 6 类：代码质量与审查（Code Quality & Review）

这类 skill 用来 **统一代码风格、审查方式和测试实践** 。

原文提到一个很有意思的例子：

> `adversarial-review` —— 生成一个"fresh-eyes"子代理来做对抗式审查，持续提出问题并迭代修复，直到剩下的只是无伤大雅的小问题（nitpicks）。

**典型例子：**

- `adversarial-review` —— 对抗式代码审查
- `code-style` —— 强制代码风格，尤其是 Claude 默认做不好的风格
- `testing-practices` —— 怎么写测试、测什么

这类 skill 可以 **作为 GitHub Action 自动运行** ，也可以作为 Claude Code 的 hooks 触发。

### 第 7 类：CI/CD 与部署（CI/CD & Deployment）

这类 skill 帮助 agent 理解 **如何构建、测试、推送、部署和回滚代码** 。

**典型例子：**

- \`babysit-pr\`\` —— 监控 PR → 重试 flaky CI → 解决合并冲突 → 开启自动合并
- `deploy-<service>` —— 构建 → 冒烟测试 → 逐步放量 + 错误率对比 → 如果回归则自动回滚
- `cherry-pick-prod` —— 独立 worktree → cherry-pick → 冲突解决 → 按模板创建 PR

这类 skill 把 **发布流程标准化** ，避免"每个人部署方式不同"的混乱。

### 第 8 类：Runbook（故障排查手册）

这类 skill 接收一个 **症状** （比如 Slack 线程、报警、错误签名），然后走完一套 **多工具调查流程** ，输出结构化报告。

**典型例子：**

- `<service>-debugging` —— 把症状映射到工具再到查询模式，针对最高流量的服务
- `oncall-runner` —— 获取报警 → 检查常见嫌疑人 → 格式化发现
- `log-correlator` —— 给定一个 request ID，从所有可能接触过的系统拉取匹配日志

这类 skill 把 **工程师的排查直觉** 编码成可执行流程。以前只有老手知道"看到这个报警先查哪三个地方"，现在 skill 告诉 AI 也这么做。

### 第 9 类：基础设施运维（Infrastructure Operations）

这类 skill 执行 **日常维护和运维程序** ，有些涉及 **破坏性操作** ，因此特别需要 **护栏和确认机制** 。

**典型例子：**

- 资源清理、依赖审批、成本调查等偏运维流程

这类 skill 的关键设计是： **必须有确认机制** 。因为涉及删除资源、修改配置等操作，不能由 AI 自动执行，必须经过人类确认。

## 三、如果把这 9 类放在一起看

你会发现一个很重要的结论：

> **skill 的适用范围覆盖了知识获取、代码生成、验证测试、业务流程、线上排障和基础设施操作，几乎贯穿了完整的软件交付链路。**

不是只有"写代码"才需要 skill。从查文档到测试验证，从故障排查到周报生成，从代码审查到部署发布——每个环节都可以用一个 skill 来标准化。

## 四、7 条关键经验：怎么写出真正有效的 skill

文章后半部分非常有价值，因为它不是在列功能，而是在总结 **如何写出真正有效的 skill** 。下面是我认为最值得记住的 7 条。

### 经验 1：不要重复常识，要补模型默认做不好的地方

如果一个 skill 只是把模型本来就知道的东西再说一遍，它的价值会很有限。

**高质量 skill 应该提供的是：**

- 组织内部知识（"我们公司内部叫 @request\_id"）
- 反常识规则（"这个函数看起来 harmless，但在高并发下会死锁"）
- 特殊约束（"所有新 API 必须返回统一的错误码格式"）
- 模型特别容易犯错的地方

文章提到一个很典型的例子：

> 前端设计 skill 的重点不是重新教模型什么是按钮和卡片，而是 **纠正模型默认会走向的审美惯性** ——比如总爱用 Inter 字体、紫色渐变、陈词滥调的布局。

### 经验 2：Gotchas 是信号最强的部分

原文明确说：

> **"任何一个 skill 里最有价值的内容，往往都是 Gotchas。也就是那些'模型最容易踩坑的点'。"**

这个判断非常重要，因为它告诉我们：

> **skill 不该是百科全书，而应该像一份"失败经验压缩包"。**

谁能把常见失败模式总结清楚，谁的 skill 就更实用。

**建议：** 在写 skill 时，专门开一个 `## Gotchas` 章节，列出 5-10 个最常见的坑。这往往比 1000 字的通用说明更有价值。

### 经验 3：善用文件系统和渐进式披露

skill 是目录而不是单文件。这意味着可以把不同粒度的信息拆开存放：

```
skill-name/
├── SKILL.md           ← 核心指令和触发条件（~100-500 token）
├── references/
│   ├── api-reference.md   ← API 说明和详细签名
│   └── gotchas.md         ← 常见陷阱
├── assets/
│   └── template.html      ← 代码模板
├── scripts/
│   └── validate.py        ← 可复用验证脚本
└── examples/
    └── sample-output.md   ← 参考样例
```

这样做的好处是： **agent 不必一上来就加载全部上下文** ，而是在任务推进过程中逐步读取所需内容。

对长任务和复杂任务来说，这种设计比单纯堆 prompt 更稳。

### 经验 4：不要把模型"钉死"在一条路径上

原文提醒要避免 **railroading** （把模型钉死在一条轨道上）。

也就是说：skill 要给模型足够的信息，但 **不要把每一步都写死** 。

因为 skill 是高度可复用的，如果说明写得太具体，它在一个场景下可能很好，但换个任务就会显得僵硬。

**好的 skill 应该：**

- 提供方向和约束
- 保留适配空间
- 让模型根据具体场景灵活调整

### 经验 5：把配置问题设计好

有些 skill 在真正使用前需要补充上下文，比如：

- Slack 发到哪个频道？
- 使用哪个 dashboard？
- 写入哪个环境？

这类信息 **不应该每次都重新解释** 。最佳做法是通过 **配置文件** 保存下来；如果没有配置，再由 agent 向用户询问。

这说明 skill 不是一次性文本，而是有\*\*"运行前准备"\*\*概念的。

**建议：** 在 skill 里放一个 `config.yml` 或 `.env` 文件，把环境相关的配置提取出来。这样同一个 skill 在不同项目里用，只需要改配置文件，不用改 skill 本身。

### 经验 6：描述字段不是简介，而是"触发条件"

这是一个 **很容易被忽视、但极其关键** 的细节。

当 Claude Code 启动时，它会先看到所有 skill 的 **name** 和 **description** ，然后决定当前任务是否该调用某个 skill。

所以 **description 的职责不是对外宣传，而是准确描述"什么情况下应该触发我"** 。

如果 description 写得模糊，再好的 skill 也可能因为 **触发不上** 而失去价值。

**反例：**

```
description: "This skill helps with web development."
```

太泛泛，Claude 不知道什么场景该用。

**正例：**

```
description: "Use this skill when the user asks to create a new React component, refactor existing JSX, or debug frontend rendering issues. Do NOT use for backend API design or database queries."
```

明确说什么场景用、什么场景不用。

### 经验 7：给模型脚本，比给模型长篇解释更有用

原文强调的一点：

> **"你能给 agent 的最强工具之一就是代码本身。"**

如果 skill 里已经放好了常用脚本、辅助函数和模板，模型就不必把轮子反复重造，而可以把精力放在 **组合和决策** 上。

这会显著提升复杂任务的成功率，也能减少样板代码带来的偏差。

**例子：**

- 不要写一大段文字解释"怎么验证 API 返回值"，而是放一个 `scripts/validate_api.py`
- 不要写一大段文字解释"怎么格式化错误日志"，而是放一个 `scripts/format_log.py`

## 五、Skill 不只是个人效率工具，还是团队知识分发机制

文章最后讨论了一个更偏组织层面的命题： **这些 skill 该如何分发？**

### 两种分发方式

| 方式 | 适合场景 | 优缺点 |
| --- | --- | --- |
| **放进代码仓库** | 项目级 skill，小团队，仓库不多 | 直接、简单；但 skill 数量膨胀后会变成额外上下文负担 |
| **做成 plugin，通过 marketplace 分发** | 大团队，跨项目复用 | 按需安装，避免上下文爆炸；但需要治理机制防止重复和劣质 skill |

### 治理问题：坏 skill 和重复 skill 非常容易产生

随着规模增大，团队会面临：

- 3 个人写了类似的 `standup-post` skill，互相冲突
- 某个 skill 的描述过时了，但还在被调用
- 某个 skill 质量很差，经常给出错误建议

原文提到一个很实用的思路： **通过 hooks 记录 skill 的实际使用情况** ，观察哪些 skill 很受欢迎、哪些 skill 明明应该被调用却经常"触发不足"。

这说明 skill 不只是要"被写出来"，还要 **被持续度量** 。

## 六、核心洞察：Skill 本质上是在做"组织经验的软件化"

如果把整篇文章压缩成更抽象的一层，我认为它讲的是一件更大的事：

> **skill 的意义，不在于给模型增加一段提示词，而在于把组织里的隐性经验转译成 agent 可以稳定复用的执行单元。**

这些经验以前可能散落在：

- 资深工程师的口头提醒里
- 某个 Wiki 页面里（但没人更新）
- 某个脚本仓库里（但没人知道怎么用）
- 某次事故复盘里（但新人没参与）
- 某个团队默认但没人写清楚的流程里

而 skill 的价值，就是把这些东西重新组织成 **agent 可以在正确时机调用的结构化能力** 。

从这个角度看，skill 更像是：

- **提示词的升级版**
- **工作流的轻量封装**
- **团队知识的可执行接口**
- **Agent 时代的新型工程资产**

## 七、一张图看懂：一个好 Skill 是如何演化出来的

```
识别高频任务或常见失败点
            │
            ▼
    选择合适的 Skill 类型
            │
            ▼
    写清触发描述 + 核心说明
            │
            ▼
    补充 gotchas + 引用资料 + 脚本 + 模板
            │
            ▼
    让 Agent 在真实任务中调用
            │
            ▼
    记录使用情况与失败案例
            │
            ▼
    迭代 Skill 内容与结构
```

**最关键的一步不是"先写很多"，而是让它先被用起来，然后围绕真实失败点持续补强。**

很多好 skill 的起点并不宏大，可能只是几行说明和一个 gotcha。但只要它解决的是高频问题、能在真实任务中被反复调用，它就会不断积累价值。

## 八、快速参考：写 Skill 的 checklist

| 检查项 | 说明 |
| --- | --- |
| description 是否明确描述了触发条件？ | 不是简介，是"什么时候该用我" |
| 是否有 Gotchas 章节？ | 模型最容易踩的 5-10 个坑 |
| 是否拆分了文件结构？ | SKILL.md + references + scripts + examples |
| 是否放了可执行脚本？ | 比长篇解释更有用 |
| 是否有配置文件？ | 环境相关参数提取到 config |
| 是否给了模型自由度？ | 不要 railroading，保留适配空间 |
| 是否包含验证机制？ | 测试 skill 对质量影响最大 |

## 九、结语：从小处开始，在使用中迭代

这篇文章最打动我的一点，是它没有把 skill 神化成某种复杂框架，而是把它还原成一种 **可以从小处开始、在使用中迭代完善** 的实践。

很多好 skill 的起点并不宏大，可能只是：

- 几行说明
- 一个 gotcha
- 一个辅助脚本

但只要它解决的是 **高频问题** 、能在真实任务中被 **反复调用** ，它就会不断积累价值，最终沉淀下来的不是文字，而是 **组织对任务、错误、流程和最佳实践的系统理解** 。

所以，与其把 skill 看成"写给模型看的说明书"，不如把它看成一个 **持续生长的能力包** 。

在一个 AI 能写代码、能测试、能部署的时代， **真正拉开差距的不是模型的智商，而是组织把经验编码成 skill 的能力** 。
